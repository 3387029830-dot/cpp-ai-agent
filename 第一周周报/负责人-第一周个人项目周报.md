### 具体完成任务列表

* **M0：CMake 最小项目骨架。**  
  完成项目初始结构搭建，建立 `CMakeLists.txt`、`src/`、`tests/` 等基础目录，使项目具备最小可编译运行能力。

* **M0.5：工程化基础配置。**  
  完成 vcpkg manifest、CMakePresets、`.clang-format`、配置目录、测试目录和忽略规则建设，为后续 C++17 工程开发提供统一环境。

* **M1：OpenAI-compatible LLM 客户端。**  
  接入 LinkAPI、DeepSeek 等 OpenAI-compatible API，支持 `.env`、系统环境变量和 `config/settings.json` 配置优先级，并补充 `/config`、`/doctor` 诊断入口。

* **M2：本地工具系统。**  
  建立 `ITool`、`ToolRegistry` 和工具 schema 机制，实现 `read_file`、`write_file`、`edit_file`、`run_command`、`list_dir` 等本地工具，并加入 workspace 路径越界防护。

* **M3：Agent 主循环。**  
  实现 `AgentLoop`：将 tools schema 发送给模型，解析模型返回的 `tool_calls`，执行工具，再把工具结果作为 `tool` 消息回填给模型，形成“模型决策 -> 工具执行 -> 结果回填 -> 再次推理”的闭环。

* **M4：安全、权限与日志。**  
  实现 `PermissionManager`，将工具风险划分为 Safe、Write、Dangerous；写入、编辑、命令执行前进行确认；实现危险命令拦截、JSONL 日志记录、`/history` 和 `/replay` 历史回放。

* **M5：交付演示增强。**  
  完成 `/demo`、`/ui`、`/skills`、`/init-env deepseek|linkapi`、配置诊断、Provider 模板和答辩文档。将 UI 从单一状态页优化为 ANSI 流式控制台风格的 Conversation / Tools / Status / Input 概览。

* **M6：Web 搜索能力。**  
  新增 `web_search` 工具和 `/search <query>` 命令。针对国内网络环境，调整为 Bing RSS 优先、DuckDuckGo 兜底，并支持 `WEB_SEARCH_PROXY_URL` 专用代理配置。后续同步修复了中文 GBK 编码导致 JSON 解析崩溃的问题。

* **M7：MCP 最小客户端。**  
  实现 `McpClient`、`McpToolAdapter` 和内置 MCP 测试 server，支持 stdio MCP initialize、tools/list、tools/call 流程。新增 `/mcp-demo`、`/mcp-call-demo`、`/mcp-connect <command> [args...]`，并支持从 `config/mcp_servers.json` 自动发现外部 MCP server 工具。

* **M8：工程化稳定性。**  
  解决新电脑构建、NMake 环境、Visual Studio generator、固定 exe 输出目录、Windows 中文命令行参数、模型名误拼、日志文件冲突、GitHub/GitLab 双平台同步等问题。新增 `AGENTS.md` 项目级规范注入，让程序启动时自动读取项目规则并追加到 system prompt。

* **SSE 流式输出升级。**

  当前 MCP 已实现 stdio 握手、工具发现、基础工具调用和外部 stdio server 接入，但 resources、prompts、SSE/HTTP transport 尚未实现。

  **原因：** 本周目标是完成课程实践可演示闭环，MCP 高级能力超出当前答辩最小范围，暂时作为后续扩展。

* **全屏 TUI 交互尚未升级。**  
  当前 `/ui` 是 ANSI 流式控制台概览，主交互仍是行式输入输出，没有实现完整全屏交互组件。

  **原因：** 当前同步阻塞式架构更适合行式流式输出；全屏 TUI 需要重构输入事件循环，短期内风险较高。

## 四、遇到的问题与困难

### 技术问题

* **GitHub 与 GitLab 多端提交混乱。**  
  本周多名组员在不同平台和不同分支提交，出现 GitHub、GitLab、本地三端历史不一致、分支冲突、MR 尚未合并等情况。

  **已尝试的解决方案：** 使用 `git fetch`、`git log --graph`、`git rev-list --left-right --count` 判断差异；根据情况采用快进、普通合并、同步分支和 MR 合并方式整理历史。

  **需要帮助或待解决：** 后续需要统一流程：组员先提交分支，负责人审核后统一合并，避免两个远端主线同时变化。

* **Windows 构建环境差异较大。**  
  新电脑上出现 `cmake` 找不到、`VCPKG_ROOT` 未设置、NMake 未加载 MSVC 环境、`cstdint` 标准库头文件找不到等问题。

  **已尝试的解决方案：** 将默认 preset 调整为 Visual Studio generator，保留 NMake 作为备用；文档中补充 vcpkg、CMakePresets、删除旧 build 目录重新配置等排错步骤。

  **需要帮助或待解决：** 答辩前需要在实际演示电脑上从零执行一次完整构建，提前排除环境问题。

* **Web 搜索在国内网络环境下不稳定。**  
  DuckDuckGo 和 Bing 搜索可能超时，中文结果还出现过 GBK 编码导致 JSON 解析崩溃的问题。

  **已尝试的解决方案：** 增加 Bing RSS 优先、DuckDuckGo 兜底链接、专用代理配置，并同步修复 GBK 到 UTF-8 编码问题。

  **需要帮助或待解决：** 现场演示需要准备离线备选方案，例如提前保存搜索结果或选择不强依赖实时联网的演示任务。

* **历史恢复 `/load` 与日志文件管理存在边界问题。**  
  `/load` 初版会列出当前刚创建的空日志，用户容易误选；多个进程同秒启动也可能造成日志文件名冲突。

  **已尝试的解决方案：** 过滤当前日志、空日志和损坏日志；日志文件名增加毫秒和进程 ID；补充历史恢复相关测试。

  **需要帮助或待解决：** 后续可进一步优化 `/history` 的标题展示和交互选择体验。

### 非技术问题

* **功能多，答辩叙事容易散。**  
  项目从 M0 到 M8 包含工程骨架、LLM、工具、安全、日志、UI、Skill、Web 搜索、MCP、AGENTS.md、跨平台构建等内容，如果逐个命令展示，会显得零散。

  **已采取的措施：** 将答辩主线整理为三层：M0-M5 核心闭环、M6-M7 扩展能力、M8 工程化稳定；同时新增 `/` 命令面板，让所有演示入口集中可见。

* **组员分工需要更清晰。**  
  随着功能增多，单个成员很难讲清全部模块。

  **已采取的措施：** 编写团队分工文档，将负责人、LLM/Agent 主循环、工具安全日志、UI/Skill/MCP/Web 搜索分成不同责任线，并对应到文档和测试文件。

### 对项目或开发流程的思考

* AI Agent 项目的价值不只是“能聊天”，而是能形成“理解用户需求 -> 模型决定工具 -> 程序安全执行 -> 结果回填 -> 继续推理 -> 记录过程”的闭环。
* 工具越多，越需要统一入口和清晰文档，否则演示时会显得碎片化；`/` 命令面板和 M0-M8 里程碑叙事对答辩非常重要。
* 安全控制是 Agent 项目的核心之一。写文件、编辑文件、执行命令都需要权限确认和 diff 预览，不能只追求自动化。
* 文档不是最后补材料，而是协作和答辩的一部分。README、测试报告、答辩提纲、团队分工必须随着代码同步更新。
* 双平台协作虽然方便，但如果缺少统一流程，会明显增加合并成本。后续需要明确“谁负责合并、何时冻结、如何同步”。
