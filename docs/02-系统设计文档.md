> 版本：v1.1

## 1. 设计目标

系统设计围绕三个目标展开：

1. 主线清晰：用户输入、模型决策、工具调用、结果回填形成闭环。
2. 模块解耦：LLM、工具、权限、日志、UI 独立演进。
3. 可验证：核心模块可以通过单元测试和手动演示验证。

## 4. 核心模块

### 4.1 Session 与 Message

职责：

- 保存 system、user、assistant、tool 四类消息。
- 为 LLM 请求提供上下文。
- 支持历史保存和回放。

```cpp
enum class Role { System, User, Assistant, Tool };

struct ToolCall {
    std::string id;
    std::string name;
    std::string arguments;
};

struct Message {
    Role role;
    std::string content;
    std::vector<ToolCall> toolCalls;
    std::string toolCallId;
    int64_t timestamp;
};
```

### 4.2 LlmClient

职责：

- 读取模型配置。
- 组装 OpenAI 兼容请求。
- 解析模型回复和 `tool_calls`。
- 对网络错误和 API 错误给出可读错误信息。

设计约束：

- API Key 从环境变量或 `.env` 读取。
- 日志中不记录完整密钥。
- 第一版使用同步请求，降低复杂度。

### 4.3 ITool 与 ToolRegistry

工具系统采用插件式设计。每个工具实现统一接口，再注册到 `ToolRegistry`。

```cpp
enum class RiskLevel { Safe, Write, Dangerous };

struct ToolResult {
    bool success;
    std::string output;
    std::string display;
};

class ITool {
public:
    virtual ~ITool() = default;
    virtual std::string name() const = 0;
    virtual std::string description() const = 0;
    virtual nlohmann::json parametersSchema() const = 0;
    virtual RiskLevel risk() const = 0;
    virtual ToolResult execute(const nlohmann::json& args) = 0;
};
```

好处：

- 新增工具时不需要修改 Agent 主循环。
- 工具可以单独测试。
- 工具风险等级可以统一交给权限模块判断。

### 4.4 PermissionManager

职责：

- 根据工具风险等级判断是否需要确认。
- 支持只读、逐次确认、本会话信任三种模式。
- 拒绝高风险命令或路径越界操作。

重点规则：

- 文件路径必须限制在工作目录内。
- 命令执行默认视为高风险。
- 删除、覆盖、批量修改必须确认。

### 4.5 Logger

职责：

- 记录用户输入。
- 记录模型请求和回复。
- 记录工具调用参数和结果。
- 保存 JSON 日志，支持历史回放。

日志建议结构：

```json
{
  "timestamp": 0,
  "type": "tool_call",
  "data": {}
}
```

## 5. Agent 主循环

```text
接收用户输入
→ 写入 Session
→ 调用 LlmClient，携带 toolsSpec
→ 如果模型直接回复，显示并结束
→ 如果模型请求工具调用
    → ToolRegistry 找工具
    → PermissionManager 检查权限
    → 执行工具
    → Logger 记录结果
    → 工具结果作为 tool 消息回填 Session
→ 再次调用模型
→ 直到无工具调用或达到最大轮次
```

最大轮次默认设置为 10，防止模型陷入无限工具调用。

## 6. 测试设计

参考课程知识库中的单元测试原则，本项目测试优先覆盖：

| 模块 | 测试重点 |
|------|----------|
| FileTools | 正常读取、文件不存在、路径越界 |
| ShellTool | 正常命令、失败命令、输出捕获 |
| ToolRegistry | 注册、查找、重复注册 |
| PermissionManager | Safe/Write/Dangerous 三类分支 |
| Logger | 写入 JSON、读取历史、字段完整 |
| ContextManager | 上下文截断和保留策略 |

测试风格采用 Given-When-Then：

```text
Given 准备输入和环境
When 执行被测函数
Then 检查结果是否符合预期
```

## 7. 代码质量设计

项目后续加入：

- `.clang-format`：统一 C++ 代码风格。
- `cppcheck`：进行静态代码检查。
- `.gitignore`：排除 build、日志、临时文件和密钥。
- 单元测试：保障重构时行为不变。

## 8. 安全设计

- API Key 不写入代码。
- 日志中对密钥、Token、认证头脱敏。
- 文件操作限制在工作目录。
- 命令执行前展示命令内容并等待确认。
- 模型输出视为不可信文本，不直接当作系统指令执行。
