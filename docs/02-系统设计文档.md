> 项目代号：cpp-ai-agent  
> 版本：v1.3
> 日期：2026-07-10

## 1. 设计目标

系统设计围绕三个目标展开：

1. 主线清晰：用户输入、模型决策、工具调用、结果回填形成闭环。
2. 模块解耦：LLM、工具、权限、日志、UI 独立演进。
3. 可验证：核心模块可以通过单元测试和手动演示验证。

## 4. 核心模块

### 4.1 Session 与 Message

职责：

- 保存 system、user、assistant、tool 四类消息。
- 为 LLM 请求提供上下文。
- 支持历史保存和回放。

```cpp
enum class Role { System, User, Assistant, Tool };

struct ToolCall {
    std::string id;
    std::string name;
    std::string arguments;
};

struct Message {
    Role role;
    std::string content;
    std::vector<ToolCall> toolCalls;
    std::string toolCallId;
    int64_t timestamp;
};
```

### 4.2 LlmClient

职责：

- 读取模型配置。
- 组装 OpenAI 兼容请求。
- 支持 SSE（Server-Sent Events）流式输出，通过 `cpr/sse.h` 原生 SSE 解析实现逐 token 推送。
- 解析模型回复和 `tool_calls`（含流式增量累积）。
- 对网络错误和 API 错误给出可读错误信息。

接口设计：

- `ILlmClient` 只有一个纯虚方法 `chatStream()`，接收可选的 `ChunkCallback` 回调。
- `chat()` 是非虚便利方法，内部调用 `chatStream()` 并传空回调（走传统 HTTP，保留 `response.text` 便于错误报告）。
- `LlmClient::configureSession()` 提取 URL/Header/SSL/Proxy 的公共配置，消除 `chatStream()` 中流式/非流式路径的重复代码。

设计约束：

- API Key 从环境变量或 `.env` 读取。
- 日志中不记录完整密钥。
- LLM HTTP 代理从 `llm.proxy_url` 或 `OPENAI_PROXY_URL` 读取；Web 搜索代理优先从 `web_search.proxy_url` 或 `WEB_SEARCH_PROXY_URL` 读取，未配置时再回退到通用代理环境变量。

### 4.3 ITool 与 ToolRegistry

工具系统采用插件式设计。每个工具实现统一接口，再注册到 `ToolRegistry`。

```cpp
enum class RiskLevel { Safe, Write, Dangerous };

struct ToolResult {
    bool success;
    std::string output;
    std::string display;
};

class ITool {
public:
    virtual ~ITool() = default;
    virtual std::string name() const = 0;
    virtual std::string description() const = 0;
    virtual nlohmann::json parametersSchema() const = 0;
    virtual RiskLevel risk() const = 0;
    virtual ToolResult execute(const nlohmann::json& args) = 0;
};
```

好处：

- 新增工具时不需要修改 Agent 主循环。
- 工具可以单独测试。
- 工具风险等级可以统一交给权限模块判断。

### 4.4 PermissionManager

职责：

- 根据工具风险等级判断是否需要确认。
- 支持只读、逐次确认、本会话信任三种模式。
- 对 `write_file` 和 `edit_file` 在确认前生成 diff preview。
- 在交互式终端中提供方向键选择 yes/no、回车确认的权限界面。
- 拒绝高风险命令或路径越界操作。

重点规则：

- 文件路径必须限制在工作目录内。
- 命令执行默认视为高风险。
- 删除、覆盖、批量修改必须确认。

### 4.5 Logger

职责：

- 记录用户输入。
- 记录模型请求和回复。
- 记录工具调用参数和结果。
- 保存 JSON 日志，支持历史回放。

日志建议结构：

```json
{
  "timestamp": 0,
  "type": "tool_call",
  "data": {}
}
```

### 4.6 UI 层

M0 阶段使用最小控制台程序验证构建链路。当前版本主界面已切换到 `src/ui/Console.*` 提供的 ANSI 增强流式控制台；非交互输出会自动关闭颜色和打字机效果。

流式输出采用事件驱动设计：AgentLoop 通过 `AgentEventCallback` 发射 `AssistantChunk`（文本增量）和 `AssistantDone`（流结束）事件，`main.cpp` 回调将事件映射为 `Console::printAssistantChunk()` / `Console::finishAssistantStream()` 调用。Console 内部管理流式状态（`streaming_` 标志），自动在首 chunk 打印 `▸ assistant` 头部、末事件打印换行尾部。当 `typewriter_` 启用时，chunk 会按 UTF-8 字符边界逐字输出（默认 8ms/字），将 API 的 burst 推送平滑为均匀字符流。

`/ui` 概览布局：

```text
┌──────────────────────────────┐
│ 对话历史                      │
├──────────────────────────────┤
│ 工具调用与输出                 │
├──────────────────────────────┤
│ 状态栏                         │
├──────────────────────────────┤
│ 输入框                         │
└──────────────────────────────┘
```

### 4.7 Skill 与 MCP 扩展

Skill 由 `config/skills.json` 配置，包含名称、说明、建议提示词、system prompt 和 `allowed_tools` 白名单。用户通过 `/use-skill <name> [target]` 激活后，AgentLoop 会注入对应 system prompt，并阻止白名单之外的工具调用。

MCP 模块提供最小 stdio 客户端：

- `McpClient`：负责 initialize、notifications/initialized、tools/list、tools/call。
- `McpToolAdapter`：把 MCP tool 包装成项目内 `ITool`，注册到 `ToolRegistry`。
- `config/mcp_servers.json`：配置外部 stdio MCP server，启动时自动发现并注册工具。

### 4.8 WebSearchTool

`WebSearchTool` 是 Safe 工具，使用 DuckDuckGo Instant Answer API 进行轻量检索，并把上游结果整理为统一文本：

- Query 和 Provider。
- Top answer，包括 title、summary、URL。
- 编号结果列表。
- 无结构化结果时给出 fallback search URL。
- 末尾给出检索结果整理建议。

### 4.9 AGENTS.md 项目规范注入

根目录 `AGENTS.md` 是项目级 Agent 规范文件，定位类似 Claude Code 的 `CLAUDE.md` 或 Cursor 的 `.cursorrules`。程序启动时会读取 workspace 根目录下的 `AGENTS.md`，并追加到 system prompt 中。

它主要承载：

- 编码规范，例如 C++17、命名风格、头文件规则。
- 项目结构认知，例如 `src/core`、`src/tools`、`tests` 的职责。
- 构建和测试约定，例如 CMake preset 和测试命令。
- 行为偏好，例如改代码前先读文件、修改后编译验证。
- 禁区规则，例如不要随意引入重型依赖或删除配置字段。

`AGENTS.md` 和 Skill 的边界：

- `AGENTS.md` 是全局项目规则，启动后对整段会话生效。
- Skill 是可切换的任务模式，通过 `/use-skill <name> [target]` 临时注入并限制工具白名单。

## 5. Agent 主循环

AgentLoop 通过 `ILlmClient::chatStream()` 调用模型，构造时通过 `enableStreaming` 参数控制是否启用 SSE 流式输出。流式模式下，每个文本增量通过 `AssistantChunk` 事件推送给 UI；无论是否有 tool_calls，响应结束时都会先发射 `AssistantDone`（流式）或 `AssistantMessage`（非流式）以闭合 UI 状态。

```text
接收用户输入
→ 写入 Session
→ 调用 LlmClient::chatStream()，携带 toolsSpec 和 ChunkCallback
  → 流式：每个 delta.content → 发射 AssistantChunk 事件
→ 发射 AssistantDone / AssistantMessage 事件（闭合 UI 流式块）
→ 如果模型直接回复，结束
→ 如果模型请求工具调用
    → ToolRegistry 找工具
    → PermissionManager 检查权限
    → 执行工具
    → Logger 记录结果
    → 工具结果作为 tool 消息回填 Session
→ 再次调用模型
→ 直到无工具调用或达到最大轮次
```

最大轮次默认设置为 10，防止模型陷入无限工具调用。

## 6. 测试设计

参考课程知识库中的单元测试原则，本项目测试优先覆盖：

| 模块 | 测试重点 |
|------|----------|
| FileTools | 正常读取、文件不存在、路径越界 |
| ShellTool | 正常命令、失败命令、输出捕获 |
| WebSearchTool | 结构化结果格式化、嵌套 RelatedTopics 展开、fallback URL 编码 |
| ToolRegistry | 注册、查找、重复注册 |
| PermissionManager | Safe/Write/Dangerous 三类分支 |
| Logger | 写入 JSON、读取历史、字段完整 |
| ContextManager | 上下文截断和保留策略 |
| SkillCatalog | 配置加载、查找、allowed_tools、target 注入 |
| McpClient | MCP 请求生成、响应解析、tools/list 和 tools/call |
| AGENTS.md 加载 | 启动时读取项目级规范并追加到 system prompt |

测试风格采用 Given-When-Then：

```text
Given 准备输入和环境
When 执行被测函数
Then 检查结果是否符合预期
```

## 7. 代码质量设计

项目后续加入：

- `.clang-format`：统一 C++ 代码风格。
- `cppcheck`：进行静态代码检查。
- `.gitignore`：排除 build、日志、临时文件和密钥。
- 单元测试：保障重构时行为不变。

## 8. 安全设计

- API Key 不写入代码。
- 日志中对密钥、Token、认证头脱敏。
- 文件操作限制在工作目录。
- 命令执行前展示命令内容并等待确认。
- 模型输出视为不可信文本，不直接当作系统指令执行。
