**四、 遇到的问题与困难**

*   **技术问题：**
    *   **GitLab Runner 环境变量全部为空**：Runner 生成的 PowerShell 环境初始化脚本存在语法错误（变量值含特殊字符未转义），导致 `${env:CI_PAGES_URL}=...` 等语句解析失败，Try/Catch 中所有变量设置被跳过。解决方案：完全不依赖 CI 环境变量，所有路径硬编码。
    *   **Artifacts 路径解析**：GitLab Runner shell executor 的 artifacts 路径相对工作目录 `C:\GitLab-Runner`，而非项目目录。首次尝试 `build/msvc-vcpkg-debug/` 时 Runner 提示 "no matching files"。解决方案：构建完成后将必要文件复制到 `C:\GitLab-Runner\build-output\`。
    *   **Artifacts 体积超限**：首次成功上传时包含 546 个文件（含 vcpkg 依赖和 .obj 中间产物），超过 GitLab 实例的上传限制（413 Request Entity Too Large）。解决方案：仅复制 `ai-agent-tests.exe` + DLLs + 一行 CTestTestfile.cmake，总体积约 10-20 MB。
    *   **YAML 与 PowerShell 语法冲突**：`'add_test(...)' | Out-File` 中的管道符 `|` 被 YAML 解析为块标量指示符；`add_test(ai-agent-tests ...)` 中的 `-` 被 PowerShell 解析为 cmdlet 名。解决方案：使用 `Set-Content` 替代管道，给参数值加双引号。
    *   **release-cli 全局 flag 顺序**：Go 的 `urfave/cli` 库要求全局 flag（`--server-url`、`--project-id`、`--private-token`）必须放在子命令 `create` 之前。错误顺序时所有 flag 失效。
    *   **Release 认证**：`CI_JOB_TOKEN` 因环境变量脚本 bug 不可用。解决方案：在 GitLab 上创建 Personal Access Token（`api` 权限），通过 `--private-token` 传入。
    *   **Release 缺少 --ref**：GitLab Releases API 要求指定 tag 指向的 commit。解决方案：`git rev-parse HEAD` 获取当前 commit SHA 传入 `--ref`。
    *   **已尝试的解决方案：** 逐轮迭代，每次只修复一个变量，通过 GitLab Pipeline 日志定位具体失败点。
    *   **需要帮助或待解决：** 无

*   **非技术问题：**
    *   **WSL2 安装失败**：DISM 组件存储损坏导致 VirtualMachinePlatform 无法启用，正在通过 `DISM /RestoreHealth` 修复（执行时间较长，已挂起）。若修复成功则可推进 SonarQube Docker 部署。

**五、 心得体会与学习收获**

*   **本周学习到的新知识/技能：**
    *   完整掌握了 GitLab CI/CD 从 Runner 注册到 Pipeline 调试、自动部署的全流程，特别是 Windows shell executor 的诸多坑点
    *   深入理解了 GitLab CI artifacts 的工作机制：路径相对工作目录、`needs:` vs `dependencies:`、体积限制
    *   掌握了 GitLab Releases API 的使用方式及 `release-cli` 工具的参数规则（Go `urfave/cli` 全局 flag 顺序要求）
    *   实践了 YAML 与 PowerShell 语法在 CI 配置文件中的交互边界问题
    *   增强了系统性排查 CI 问题的能力：逐轮迭代、每次只改一个变量、通过日志精确锁定根因

*   **对项目或开发流程的思考：**
    *   CI 配置文件的调试反馈周期太长（push → 等待 Runner 执行 → 查看日志），每轮约 3-5 分钟。后续可以通过 GitLab CI Lint 工具在 push 前验证语法
    *   Shell executor 的灵活性与复杂性并存——对单项目专用 Runner 足够，但多项目共享时应考虑 Docker executor
    *   自动部署将"构建 → 测试 → 发布"串联为全自动流程，消除了人工打包上传可能出现的遗漏

*   **自我改进方面：**
    *   在 CI 配置中遇到环境差异时，优先考虑"硬编码 + 注释说明"的策略，而非依赖可能不可用的运行时变量
    *   注意 YAML 与目标脚本语言的语法边界，避免隐式冲突
    *   Deploy 阶段应关注发布包的用户体验——不仅包含可执行文件，还要包含配置文件模板和说明文档
