**四、 遇到的问题与困难**

*   **技术问题：**
    *   **GitLab Runner prepare environment 失败（耗时最长，约 6 小时）**：Runner shell executor 通过 `su` 切换用户时 session 瞬间关闭，排查涵盖 Runner 日志、strace 系统调用跟踪、systemd service 配置、PAM 认证等多个层面。最终放弃 `su` 中间层，Runner 直接以 root 身份执行 job。
    *   **SonarQube Docker 镜像拉取被墙**：Docker Hub 默认 registry 从 VM 无法访问（i/o timeout）。解决方案：配置 `registry-mirrors: ["https://docker.m.daocloud.io"]` 通过 DaoCloud 加速。
    *   **SonarQube 10.3 旧数据卷权限冲突**：8.3 的 data volume 挂载到 10.3 后 ES8 无法创建目录（AccessDeniedException: /opt/sonarqube/data/es8）。解决方案：创建全新 volume。
    *   **GitLab CI YAML 与特殊字符冲突**：花括号 `{}`、冒号空格 `: `、UTF-8 BOM 在 GitLab YAML 解析器中均会触发 "mapping values are not allowed" 错误。解决方案：逐个排除 → 最终全文件纯 ASCII 化。
    *   **VM 到校内 GitLab 网络不稳定**：`git pull`/`git clone` 从 VM 到 `csgitlab.whu.edu.cn` 频繁超时（Could not connect to server）。临时绕过方案：脚本直接从公网 CDN 下载，Runner 自行 fetch 代码（机制不同，偶能通过）。
    *   **Artifacts 下载 403 Forbidden**：shell executor 的 artifacts 在本地文件系统，但 GitLab 仍尝试通过 HTTP 下载。解决方案：`needs` 添加 `artifacts: false`。

*   **非技术问题：**
    *   GitLab 服务器（校内 `csgitlab.whu.edu.cn`）稳定性欠佳，部分时段无法连接，影响 Pipeline 触发和代码拉取。
    *   VMware 虚拟机网络环境受限（NAT 网段 192.168.149.x vs Bridge 10.128.x.x），组员无法直接访问 VM，需通过 Windows netsh portproxy 转发 Jenkins/SonarQube 端口。

*   **对项目或开发流程的思考：**
    *   CI/CD 配置的调试反馈周期过长（push → 等待 Runner → 查看日志 → 修 → push），尤其在 GitLab 服务器不稳定时更甚。后续可以通过 `gitlab-ci-local` 或 `act` 在本地模拟运行
    *   SonarQube Community Edition 对 C++ 的支持有限，对于预算紧张的学术项目，cppcheck + CI artifacts 是务实的替代方案——虽然缺少 Web Dashboard，但功能完整且零成本
    *   Linux shell executor 的灵活性虽高，但缺乏 Docker executor 的环境隔离性——后续若项目规模扩大，应考虑迁移到 Docker executor

*   **自我改进方面：**
    *   排查 CI 问题时逐步迭代、每次只改一个变量的策略非常有效——15 轮 Pipeline 调试无一次是盲目修改
    *   当第三方插件兼容性反复出问题时，果断切换到更简单的独立工具（SonarQube → cppcheck），避免在排错上投入更多时间
    *   注意 CI 配置文件的编码和字符集问题——Windows 编辑器自动添加 BOM 是 CI YAML 解析失败的常见根因
