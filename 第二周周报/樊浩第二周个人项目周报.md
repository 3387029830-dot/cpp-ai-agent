  **原因：** GitLab `main` 是保护分支，本地无法直接推送到 `main`。目前已将修复推送到同步分支，最终合并需要在 GitLab 页面完成 Merge Request 操作。

  **原因：** 虽然已经完成可用的双模式界面，但当前界面还在根据答辩展示效果持续打磨。需要在真实投屏环境下检查字号、间距、对比度、页面层次和操作流畅度，避免现场显得粗糙。

* **Web 搜索现场稳定性仍需准备备选方案。**  
  **原因：** 国内网络环境和代理状态不稳定，Bing RSS、DuckDuckGo 等入口可能超时。虽然项目已支持代理和兜底链接，但正式答辩仍应准备离线结果或不依赖实时搜索的备用演示路径。

* **MCP 高级协议能力暂未实现。**  
  **原因：** 当前 MCP 已覆盖 stdio 握手、工具发现、工具调用和 AgentLoop 工具适配，已经满足课程演示的最小客户端目标。resources、prompts、SSE/HTTP transport 属于扩展边界，不纳入本次两周实训交付范围。

## 四、遇到的问题与困难

### 技术问题

* **Web UI 初版交互不完整。**  
  初版 Web 页面中左侧 `Tools`、`Skills` 更像静态导航，点击后没有明确切换主内容；右侧 Activity 也容易出现重复事件，导致用户不理解它的作用。

  **已尝试的解决方案：** 修改前端结构，新增主视图切换逻辑，让 `Tools / Skills / Runs` 进入真实内容页；Activity 改成运行事件时间线，并通过事件 ID 去重；增加 hash 深链支持，例如 `/#tools`、`/#skills`、`/#runs` 可以直接打开对应视图。

  **收尾处理：** 本周期内已完成一次视觉方向重构，减少硬边框和卡片堆叠；最终展示时以当前可用版本为准，不再扩展新的交互功能。

* **GitHub push protection 拦截 GitLab 旧历史。**  
  GitLab 历史中曾出现旧 token 痕迹，即使当前文件已经改成环境变量，GitHub 在推送整段历史时仍会扫描旧提交并拒绝。

  **已尝试的解决方案：** 不直接推送 GitLab 历史到 GitHub，而是在 GitHub 干净主线上用文件快照方式同步最新内容，避免带入敏感历史。GitLab 侧继续走 MR 分支。

  **收尾处理：** 已采用文件快照同步方式绕开旧历史问题；该问题作为团队安全经验沉淀：任何 token 都不得写入提交历史，应统一使用 GitLab/GitHub 的环境变量或 secrets 管理。

* **GitLab CI job rules 与 needs 不匹配。**  
  `build-linux` 被 rules 跳过时，`test-linux` 仍然声明强依赖，导致流水线在创建阶段失败。

  **已尝试的解决方案：** 为 `test-linux` 和 `deploy-linux` 增加与 `build-linux` 一致的 rules，并将相关 needs 改成 `optional: true`。同时用 Python YAML 解析检查 `.gitlab-ci.yml` 结构有效。

  **收尾处理：** 本项目周期内已完成当前 CI 规则修复；相关经验沉淀为维护原则：新增可选 job 时必须同步检查 rules 和 needs，避免再次出现“job 不存在但被依赖”的问题。

* **答辩功能多，演示容易显得零散。**  
  项目已有 LLM、AgentLoop、Tools、Permission、History、Skill、MCP、Web Search、AGENTS.md、SonarQube、Web Control Deck、CI/CD 等能力，如果逐条展示命令，会让听众难以抓住主线。

  **已尝试的解决方案：** 将项目划分为 M0-M5 核心 Agent 闭环、M6-M7 扩展能力、M8-M10 工程化与演示增强三层，并在 Web Control Deck 中增加 Chat、Tools、Skills、Runs 导航，帮助答辩时聚合展示。

  **收尾处理：** 已将该问题纳入最终答辩脚本整理：演示时按“核心闭环 -> 扩展能力 -> 工程化交付”三层展开，避免逐条命令堆叠。

### 非技术问题

* **多人提交导致远端状态复杂。**  
  第二周组员在 GitHub、GitLab、不同分支上持续提交，远端主线和 MR 状态变化较快，负责人需要不断确认最新版本是否被覆盖。

  **已采取的措施：** 通过 `git fetch`、`git log`、`git status`、远端分支比较和 MR 分支同步来确认状态；对 GitLab 保护分支统一采用 MR 方式，避免直接强推；同步后检查本地与远端文件树是否一致。

* **界面审美和实际使用预期不一致。**  
  在 Web Control Deck 初版完成后，发现虽然功能存在，但界面显得生硬，尤其是方框过多、层级不够自然，不符合答辩展示对“高级感”的要求。

  **已采取的措施：** 重新调整视觉语言，减少硬边框和卡片堆叠，采用暗色大画布、侧边索引、输入坞和更弱的边界表达，让页面更像一个工作台而不是简单状态页。

## 六、心得体会与学习收获

### 本周学习到的新知识/技能

* 学习并实践了 Web UI 与本地 C++ Agent 后端的轻量集成方式，理解了本机 HTTP 服务如何把 Agent 事件、权限确认、工具列表和状态信息暴露给浏览器。
* 进一步理解了 GitLab CI 中 `rules`、`needs`、`optional` 的关系，认识到流水线是否能创建不仅取决于脚本能否运行，还取决于 job 图本身是否自洽。
* 对 GitHub/GitLab 双远端协作有了更深入的经验，尤其是当两个平台历史不一致、保护分支限制、push protection 拦截同时出现时，不能简单依赖一次 merge 解决问题。
* 了解了 SonarQube 质量评估接入方式，认识到 DevOps 不只是自动构建和测试，也包括质量门禁、静态分析和安全配置。

### 对项目或开发流程的思考

* 项目负责人需要关注的不只是“功能有没有做出来”，还要关注“功能能不能被别人理解、能不能在答辩现场稳定展示、能不能被文档和测试支撑”。第二周的工作让我更明显地感受到，演示体验、仓库状态、CI 稳定性和文档完整度都会影响项目最终质量。
* Web Control Deck 的迭代让我认识到，界面不是把按钮和数据摆上去就结束。真正可用的界面应该让用户知道自己在哪里、能做什么、下一步会发生什么。Tools、Skills、Runs 这类导航如果没有真实内容，就会损害用户信任。
* GitHub/GitLab 同步问题说明，多人协作项目必须有统一的分支和合并规范。否则后期负责人需要花大量时间恢复主线一致性，影响功能开发和答辩准备。

### 自我改进方面

* 通过本次实训，我认识到今后在项目初期就应更早设定演示验收标准，尤其是 Web UI、PPT、答辩脚本这类对外展示内容，不能等到最后才根据观感反复调整。
* 在 CI/CD 和版本管理方面，应提前建立更严格的规则：敏感信息一律进环境变量，主分支不直接开发，合并前必须检查 CI 和文档影响。
* 在负责人沟通方面，需要继续把“技术进展”翻译成“答辩可讲的项目价值”，让组员负责的模块都能自然接入整体故事，而不是各讲各的。
